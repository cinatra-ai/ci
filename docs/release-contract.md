# The cinatra-ai release contract

One page: what a tagged release is, what it carries per repo type, and which
gates enforce it. The reusable release workflows live in
[`cinatra-ai/.github`](https://github.com/cinatra-ai/.github) (see its
[`examples/`](https://github.com/cinatra-ai/.github/tree/main/examples) for
copy-me callers); this page is the contract they implement.

## Tag rules (all repos)

- **Owners cut tags; CI only decorates them.** No workflow creates a tag
  (`gh release create --verify-tag` fails closed on a missing tag), and no
  workflow overwrites author-written release notes ‚Äî an existing Release body
  is backfilled only when empty.
- **A release tag is an annotated semver tag**: `v<MAJOR>.<MINOR>[.<PATCH>]`
  with optional prerelease/build suffix, created with `git tag -a`. Lightweight
  semver tags fail the release workflows (and the marketplace deploy pipeline)
  closed.
- **A non-semver `v*` tag is a checkpoint/marker**: every release workflow
  treats it as a no-op ‚Äî use one deliberately when a tag must NOT release or
  deploy.
- **Every semver tag gets a GitHub Release page with auto-generated
  "What's Changed" PR-list notes**, so any tag maps to the exact set of
  shipped PRs. Backfilling Release pages for historical tags is an owner
  decision (the thin callers expose a `workflow_dispatch` input for it), never
  an automatic act.

## What a release carries, per repo type

| Repo type | Release path | The Release page carries |
|---|---|---|
| **npm-published extensions** (the `@cinatra-ai/*` connector/agent/skill repos) | thin `release: published` caller ‚Üí `reusable-extension-release.yml` in `cinatra-ai/.github` (typecheck/test ‚Üí packlist gate ‚Üí `npm pack` ‚Üí provenance attestation ‚Üí marketplace publish-proxy submit; never a direct registry publish) | PR-list notes + the **exact packed tarball** the marketplace received + `SHA256SUMS.txt`, plus GitHub's auto source archives |
| **cinatra core** | `create-release` job in its image-build workflow (tag-triggered, semver-gated) | PR-list notes + auto source archives ‚Äî the archive is deliberately **faithful to the tag** (private app; no `export-ignore` trimming beyond confirmed-secret paths) |
| **marketplace** (PHP, tag-deployed) | its own deploy pipeline (semver gate ‚Üí preflight ‚Üí annotated-tag check ‚Üí dispatch to ops) **plus** the additive `github-release.yml` caller below ‚Äî the Release page is deploy-inert; deploys only ever flow through the dispatch | PR-list notes + auto source archives |
| **Non-extension repos that cut tags** (PHP plugin repos, the CI/workflow repos, ‚Ä¶) | thin `github-release.yml` caller ‚Üí `reusable-release-notes.yml` in `cinatra-ai/.github`, pinned to a commit SHA | PR-list notes + auto source archives |
| **Repos that do not tag** (docs, website, internal trackers, design, ‚Ä¶) | none ‚Äî **adopt-on-first-tag**: before cutting a first `v*` tag, add the thin caller (copy `examples/github-release.yml` from `cinatra-ai/.github`) and a root `.gitattributes` (below) | ‚Äî |
| **ops** | tags are operational markers, not releases; ops deploys via `repository_dispatch`/cron/manual only | ‚Äî (opt in deliberately if that ever changes) |

## Source archives: `.gitattributes export-ignore`

GitHub's auto `Source code (zip/tar.gz)` on every Release is generated by
`git archive`, so a root `.gitattributes` with `export-ignore` rules controls
what it contains. The convention (already on `main` of every npm-published
extension repo):

- exclude CI/internal noise with the recursive form ‚Äî `.github/** export-ignore`
  (a bare directory pattern does **not** recurse reliably for `export-ignore`),
  plus git plumbing (`.gitattributes`, `.gitignore`);
- keep build/tooling config in the **source** archive for reproducibility
  (it is excluded from the npm tarball instead);
- the rules only affect **tags cut after** the `.gitattributes` landed ‚Äî
  adding it never rewrites an existing tag's archive. Verify with
  `git archive --format=tar.gz <tag> | tar tzf -`;
- GitHub does not guarantee checksum stability of auto-archives. Anything that
  must be hash-pinned is an **uploaded release asset** (the extension pipeline
  uploads the exact tarball + `SHA256SUMS.txt` for this reason);
- cinatra core stays untrimmed by decision (fidelity-to-tag beats trimming for
  a private app).

## npm tarballs: the `files` allowlist + the packlist gate

For published extension packages the distribution artifact is the `npm pack`
tarball, and it crosses a trust boundary ‚Äî so it ships a **minimal allowlist**,
not a trimmed denylist:

- every npm-published extension repo declares a `package.json` `"files"`
  allowlist (fails closed: new files are excluded by default). `.npmignore` is
  deliberately **not** used ‚Äî it fails open and silently replaces `.gitignore`
  when present. npm force-includes `package.json`/`README*`/`LICENSE*`; the
  allowlist must include the package's real entry points (`main`/`types`/
  `exports` targets ‚Äî several extensions ship `src/` directly);
- the **packlist gate** in `reusable-extension-release.yml` fails the release
  if `npm pack --dry-run --json` lists any internal path (`.github/`,
  `.planning`, env files or key material, tests/fixtures, CI/build/lint
  tooling config). Local check: run the same dry-run before tagging;
- the gate enforces on a repo's releases once the repo's thin caller pins a
  `cinatra-ai/.github` commit that contains the gate. Caller pins are
  normalized opportunistically ‚Äî when touching a caller, bump its pin (and
  the `# v0.0.0-main.<short-sha>` comment) to current `main`.

## Wiring a repo (thin callers)

Copy from `cinatra-ai/.github` `examples/` and pin `uses:` to a 40-char commit
SHA (org pin policy; the `actions-pinned-gate` rejects mutable refs):

- extension repos: `examples/release.yml` (needs `contents: write`,
  `id-token: write`, `attestations: write`, `secrets: inherit` for the
  org submit token),
- everything else that tags: `examples/github-release.yml` (needs
  `contents: write` and **no secrets**).

The canonical sources for the extraction tooling's templates are the live org
files; the copies in the extension-release-tooling repo are reference mirrors
kept byte-identical to them.

## The release-approval wall ‚Äî pin a GATED reusable-release ref

A tagged extension release must **pause for a human approval** before it
publishes. That pause is a GitHub `release-approval` Environment (required
reviewer) declared on the `release` job of `reusable-extension-release.yml`
(`environment: release-approval`). The wall first shipped at
`reusable-extension-release.yml@1e4448a‚Ä¶` (tag `v0.1.1`); the older
`fea09b38‚Ä¶`/`671d3e6f‚Ä¶` (`v0.1.0`) refs have **no** `environment:` on that job.

**Requirement (fails closed):** every extension repo's `release.yml` MUST pin
`reusable-extension-release.yml@<ref>` to a **gated ref** ‚Äî a SHA in
[`config/release-workflow-gated-refs.json`](../config/release-workflow-gated-refs.json).
Pinning an ungated ref publishes with **no** approval pause.

This is **not** an opt-in convenience ‚Äî it is enforced by the
**release-workflow-pin-drift-gate** (a scheduled org-wide scan that reds if any
repo pins a non-gated ref). The lesson it encodes: a control that is
*opt-in-per-repo* (each repo must remember to pin the gated ref) fails **open**
for every repo that did not opt in ‚Äî exactly the fail-open-by-default trap of an
auth check gated on a per-repo flag. The durable fix is enforced-by-default: a
default that fails closed plus a gate that proves every repo is on it. When a
new gated reusable-workflow tag is published, verify its `release` job still
carries `environment: release-approval`, add its SHA to the allowlist, and bump
the extension repos onto it.
